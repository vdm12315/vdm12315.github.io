Nacos什么是Nacos一

1、什么是Nacos

一个更易于构建云原生应用的动态服务发现、服务配置和服务管理平台。

2、注册中心

  • 上游服务向注册中心注册地址，供下游服务调用。在服务众多的微服务场景中，频繁的服务创建和销毁会导致上游服务发现不了下游服务，注册中心就是解决这个问题的。

  • 注册中心会和服务维护一个心跳信息，当一定时间内没有收到心跳信息，那么在注册中心中把该服务标致为down

minio

Minio-一

1、MinIO

  • MinIO是一款高性能、分布式、基于Apache License v2.0开源协议的对象存储服务。适合存储大容量非结构化数据，例如：视频、图片、日志文件、备份数据等。

2、MinIO基础概念

  • Object：存储到MinIO的基本对象，如：文本、视频等

  • Bucket：用来存储Object的逻辑空间。每个Bucket之间的数据是相互隔离的。对于客户端而言，就相当于一个存放文件的顶层文件夹。

  • Drive：即存储数据的磁盘，在MinIOn启动时，以参数的方式传入。MinIOn中所有的对象数据都存储在Drive中。

  • Set：即一组Drive的集合，分布式部署根据集群规划会自动划分一个或多个Set，每个Set中的Drive分布在不同的位置。一个对象存储在一个Set上。

    • 一个对象存储在一个Set上

    • 一个集群划分多个Set

    • 一个Set包含的Drive数量是固定的，默认由系统根据集群规模自动计算得出

    • 一个Set中的Drive尽可能分布在不同的节点上

3、纠删码EC（Erasure Code）

  • 纠删码是一种恢复丢失和损坏数据的数学算法，MinIO采用Reed-Solomon code将对象拆分成N/2数据和N/2奇偶校验块。这就意味着如果是12块盘，一个对象会被分成6个数据块、6个奇偶校验快，你可以丢失任意6块盘，仍可以从剩下的盘中将数据恢复。

4、MinIOn分布式

  • 一般在部署分布式MinIO的时候，要求盘符为4或4的倍数

kafka

Kafka生产者(六)

1、Kafka生产者同步和异步消息（生产者配置）

  • 同步：生产者向broker发送消息，等broker返回成功之前，生产者一直阻塞等到broker返回才进行下一步的动作。

  • 异步：生产者向broker发送消息，未等broker返回成功，就异步执行下一步动作。

  • 以上大多使用同步消息，在业务侧使用代码实现。异步并未提升太大的性能，同时带来了消息可能丢失的风险

2、Kafka的ack配置（生产者配置）

  • ack=0 #Kafka-cluster不需要任何broker收到消息，就立即返回ack给生产者，此方式最容易丢消息，性能最好。

  • ack=1（默认） #多副本之间的leader已经收到消息，并发消息写入本地log中，才会返回ack给生产者，此方式最平衡。

  • ack=-1/all #多副本之间至少有min.insync.replicas台机器收到消息并写入本地，才返回ack给生产者，建议min.insync.replicas配置大于等于2.这样至少有一个leader和一个follower，当leader挂了的时候，该follwer可以顶替成leader。改模式性能最差，安全性最高。

3、生产者发送缓冲区

  • Kafka默认会创建一个消息缓冲区，用来存放要发送的消息，缓冲区默认32m（可配置）

  • Kafka本地线程会去缓冲区一次拉16k（可配置）的数据发送到broker

  • 如果缓冲区的数据没有16k大小，那么间隔10ms（可配置）也会将已拉到的数据发送到broker

3、副本的概念

副本是对分区的备份。在集群环境中，不同的副本会被部署在不同的broker上。下面例子：创建1个主题，2个分区，3个副本

  • Topic：topic名字

  • Partition：分区

  • Leader：负责读和写,也就是生产者和消费者都与leader通信，同时会把数据同步到foolower。Partition: 0 Leader: 1（表示0号分区在brokerid为1的broker节点上）

  • Replicas：broker的节点数

  • Isr：可以同步和已同步的节点会被存入isr集合中。如果isr中某个节点性能过差，会被踢出Isr集合中。该字段可以判断broker节点同步是否正常

5、关于分区消费组消费者细节

图中Kafka集群有两个broker，每个broker中有多个partition。

  • 一个partition只能被一个消费组里的某一个消费者消费，从而保证消费顺序，Kafka只在partition的范围内保证消息消费的局部顺序性，但是多个partition的多个消费者消费的总的顺序性是得不到保证的。

  • partition的数量决定了消费组中消费者的数量，建议同一个消费组中消费者的数量不要超过partition的数量，否则多出来的消费者消费不到消息

  • 如果消费者挂了，那么会触发rebalance机制，会让其他消费者来消费该分区

  • 一个partition只能被一个组内的一个消费者消费，但是一个消费者可以消费多个partition

Kafka消息类别（四）

1、单播消息

如果多个消费者在同一个消费组里，那么只有一个消费者可以收到订阅的topic中的消息，也就是同一个消费组中，只有一个消费组可以收到消息。

2、多播消息

不同的消费组订阅同一个topic，每个不同的消费组里有且仅有一个消费者能收到消息。

3、查看消费组及信息

  • GROUP：组名

  • TOPIC：topic名

  • PARTITION： 分区

  • CURRENT-OFFSET：当前消费偏移量

  • LOG-END-OFFSET： topic中消息总量

  • LAG：当前有多少消息未被消费，也就是堆积

  • HOST：Kafka服务器

4、Kafka中的主题和分区

1、主题Topic

  • 主题-topic在kafka中是一个逻辑概念，kafka通过topic将消息进行分类，不同的topic会被订阅该topic的消费者消费

2、分区

  • 为了解决某个topic消息过大的问题（比如几个T的消息文件），引入了分区的概念，通过将消息进行分布式存储，减小每个分区的大小。比如：1个分区1 90G，那用三个分区来存储，每个分区只有30G

  • 同时，分区可以分担读和写的负载

  • __consumer_offsets-49：kafka内部创建了__consumer_offsets，包含50个分区。这个主题用来存放消费者的某个主题的便宜量，每个消费者会把消费的主题偏移量自主上报给kafka的默认主题。kafka为了提升整个主题的并发性，默认设置了50个分区，可以通过参数调整

  • 提交到50个主题中的哪个分区，通过hash(consumerGroup)%__consumer_offers主题分区数

  • 提交到该主题中的内容是：key是consumerGroupid+topic+分区号，value就是当前offset值
